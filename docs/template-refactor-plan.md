# 模板工程裁剪改造计划

## 1. 背景

当前项目仍以 OpenFlare 完整业务产品为基础进行设计和实现，`openflare_server` 与 `openflare_agent` 围绕节点注册、心跳、同步、版本分发、OpenResty 配置渲染、代理规则管理和观测分析展开。

本次改造目标不是继续增强 OpenFlare，而是将项目裁剪为一个可复用的模板工程，保留通用底座能力，移除大部分 OpenFlare 强业务耦合内容，并同步完成工程规范化与目录结构重组。

本次改造重点集中在 `server`，`agent` 最终整体删除。

## 2. 改造目标

模板工程保留以下基础模块：

* 用户模块
* 版本升级模块
* 邮箱模块
* 文件上传模块
* 安全模块

模板工程需要同时满足以下目标：

* 服务端可独立启动，不依赖 Agent、节点和 OpenResty
* 数据模型、API、前端菜单、Swagger、配置项与模板目标一致
* 接口与界面同步移除、同步收敛，避免残留入口
* 服务端严格遵循 MVC 架构开发
* 项目目录结构同步规范化，逐步迁移到标准 Go 工程布局

目标目录结构：

```text
project/
├── cmd/                    # 应用入口
├── internal/               # 私有包
│   ├── handler/            # HTTP 处理器（Controller）
│   ├── service/            # 业务逻辑层
│   ├── repository/         # 数据访问层
│   ├── model/              # 领域模型/数据模型
│   ├── dto/                # 数据传输对象
│   ├── middleware/         # 中间件
│   └── pkg/                # 项目内公共包
│       ├── utils/          # 工具函数
│       ├── database/       # 数据库封装
│       └── ...             # 其他业务无关组件
└── pkg/                    # 公开包（可选）
```

## 3. 当前系统分析

### 3.1 可保留的基础能力

1. 用户与认证能力
   当前已具备注册、登录、Session、Token、角色与状态控制，具备模板化复用价值。

2. 邮箱能力
   当前已具备邮箱验证码、密码重置、邮箱绑定等能力，可整理后保留。

3. 文件上传能力
   当前已有上传、删除、下载、元数据入库链路，可作为模板文件模块基础。

4. 安全基础设施
   当前已有密码散列、验证码、随机码、鉴权中间件、限流和 Turnstile 校验等能力。

5. 服务端版本升级能力
   当前已有服务端自升级链路，可保留为模板工程运维能力。

### 3.2 需要移除的 OpenFlare 业务能力

* Agent 子系统
* 节点管理与心跳同步
* 配置版本发布与分发
* OpenResty 代理规则
* 域名与证书分发
* 访问日志、观测分析、看板

### 3.3 当前主要问题

1. 业务边界仍然是 OpenFlare
   现有设计、规范、部署说明、配置说明和 Swagger 仍以 OpenFlare 为主。

2. 数据模型耦合严重
   当前数据库模型与迁移逻辑默认包含大量 OpenFlare 专属实体。

3. 路由与前端入口耦合严重
   接口、页面、导航、API client、类型定义都和 OpenFlare 业务深度绑定。

4. 分层职责仍有混杂
   虽已有 `controller`、`service`、`model` 目录，但历史代码里仍存在职责边界不清的问题。

5. 目录结构仍为历史平铺形态
   服务端目录未按标准 Go 工程布局组织，不利于模板工程长期维护和扩展。

## 4. 改造原则

1. 先改文档，再改代码
   当前目标已经超出原有设计边界，正式编码前必须先完成文档重定义。

2. 先重定义边界，再做裁剪
   先明确模板工程保留什么，再处理模型、路由、页面、配置项和目录迁移。

3. 接口与界面同步移除
   删除任一 OpenFlare 业务能力时，必须同步移除后端接口、前端页面、导航入口、API client、类型定义、Swagger 和相关文档。

4. 服务端严格遵循 MVC
   Controller 负责请求处理与响应封装，Service 负责业务编排，Repository 负责数据访问，Model 负责领域/数据模型表达。

5. 目录结构调整与分层调整同步推进
   本次不是只在旧目录上做删减，而是要同步迁移到 `cmd + internal + pkg` 结构，避免后续二次返工。

6. 保守处理数据库迁移
   不允许简单依赖 `AutoMigrate` 掩盖结构变化，必须显式处理数据库版本与迁移路径。

## 5. 分阶段实施计划

### 阶段一：文档与目标边界重定义

目标：

* 将产品定位从 OpenFlare 控制面调整为通用模板工程
* 明确保留模块、移除模块、MVC 约束和目录结构目标

任务：

* 更新 `docs/design.md`
* 更新 `docs/development-guidelines.md`
* 更新 `docs/development-plan.md`
* 更新 `docs/deployment.md`
* 更新 `docs/app-config.md`
* 视需要更新 `README.md`

验收标准：

* 文档不再以 Agent、节点、OpenResty、配置分发为系统主干
* 文档明确目录结构调整也属于本次主线

### 阶段二：后端模型与配置瘦身

目标：

* 让服务端核心模型只保留模板需要的实体

任务：

* 从现有注册模型中移除 OpenFlare 专属模型
* 重构数据库 schema 版本与迁移逻辑
* 精简配置项和运行时 Option 逻辑
* 去除节点、OpenResty、GeoIP、观测相关配置
* 同步设计 `model/repository/dto` 的迁移边界

验收标准：

* 数据库初始化只创建模板需要的表
* 服务端运行配置不再暴露 OpenFlare 专属能力

### 阶段三：后端接口裁剪与 MVC 收敛

目标：

* 让后端只暴露模板保留能力，并完成核心模块分层整理

任务：

* 删除节点、Agent、代理、证书、配置分发、观测相关接口
* 保留并整理 `user`、`update`、`file`、邮箱、安全相关接口
* 清理无效测试、Swagger 注释和服务依赖
* 建立接口删除与前端删除的同步清单
* 将保留模块按 `handler/service/repository/model/dto` 进行拆分

验收标准：

* 路由中仅保留模板能力相关接口
* 服务端编译通过
* Swagger 只包含保留接口

### 阶段四：前端管理端同步裁剪

目标：

* 前端只保留模板工程相关页面与导航

任务：

* 删除节点、网站、证书、性能、日志、配置版本等 feature
* 保留登录、注册、找回密码、用户管理、系统设置、文件管理、升级相关页面
* 收敛导航、API client、类型定义、测试用例
* 与阶段三一一对应，后端接口移除与前端界面移除必须同批次完成

验收标准：

* 前端菜单与页面只包含模板能力
* 不存在失效导航和失效请求

### 阶段五：目录结构重组

目标：

* 将服务端目录迁移到标准 Go 工程结构

任务：

* 建立 `cmd/` 作为应用入口
* 建立 `internal/handler`
* 建立 `internal/service`
* 建立 `internal/repository`
* 建立 `internal/model`
* 建立 `internal/dto`
* 建立 `internal/middleware`
* 将项目内公共能力收敛到 `internal/pkg`
* 视需要保留或启用 `pkg/` 作为公开包目录
* 调整 import、初始化流程和测试路径

验收标准：

* 服务端主目录符合目标结构
* 保留模块已迁移到新目录
* 历史平铺目录不再继续扩张

### 阶段六：清理 Agent 与部署残留

目标：

* 从仓库中彻底移除 Agent 及其部署联调残留

任务：

* 删除 `openflare_agent`
* 清理脚本、Docker、README、部署文档、联调说明中的 Agent 内容
* 清理测试命令和 CI 中的 Agent 引用

验收标准：

* 仓库中不再存在 Agent 运行、部署、升级、联调入口

### 阶段七：回归验证与模板固化

目标：

* 确保模板工程可稳定作为新项目起点使用

任务：

* 执行后端测试、前端构建和必要回归
* 验证登录、注册、密码重置、邮箱验证码、文件上传、版本升级主链路
* 验证接口删除和界面删除是否同步完成
* 补充缺失测试与说明

验收标准：

* 服务端启动成功
* 前端构建成功
* 保留模块可用
* 文档、代码、目录结构、配置项保持一致

## 6. 主要风险

1. 数据库迁移风险
   删除历史业务模型后，旧库升级和空库初始化都可能受到影响。

2. 前后端不同步风险
   如果接口删除后前端未同步收敛，会产生大量运行时错误和残留入口。

3. 工程规范化落空风险
   如果只删业务代码而不同步收敛 MVC 分层，模板工程仍会继承历史结构问题。

4. 目录迁移风险
   目录结构重组会牵动入口、依赖、初始化和测试路径，如迁移顺序不当，容易产生大面积编译错误。

5. 升级模块耦合风险
   服务端升级链路可能仍包含 OpenFlare 命名或路径依赖，需要单独梳理。

6. 文档滞后风险
   如果代码先变、文档后补，后续实施标准容易失真。

## 7. 建议实施顺序

1. 先完成设计、规范、计划文档重定义
2. 再做模型、配置和后端接口瘦身
3. 同步推进前端界面与导航裁剪
4. 再迁移目录结构到 `cmd/internal/pkg`
5. 最后删除 Agent 和部署残留
6. 结束前统一做回归与文档收口

本文件作为“OpenFlare 裁剪为模板工程”的专项计划文档使用。

后续如出现以下变化，应同步更新本文件：

* 保留模块清单变化
* 删除模块清单变化
* MVC 分层规则变化
* 目录结构目标变化
* 阶段目标与实施顺序变化
